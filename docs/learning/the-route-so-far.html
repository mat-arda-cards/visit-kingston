The Route So Far

Three connected upgrades — the team calls them "epics" — to the Kingston ferry app. Explained the way you'd explain actually running a ferry terminal, because the metaphor turns out to fit disturbingly well.

One analogy per epic — the everyday version
Three steps — what actually happened, in order
One pop quiz — to check it landed
Berth 1

Dockside Safety Sweep

E01 — Security hardening on the live app

The app was already live and open to the public — anyone could visit it, no account needed. A safety review found eight real problems: things a stranger could use to fill up the server's storage, trick a visitor into a scam link, sneak a malicious link onto a public page, crash the operator's dashboard with one bad file, and show event times seven hours off, among others. Every fix shipped separately, each with its own automatic proof that it worked, and the site never went offline for a second.

Terms, translated
Anonymous
— no login required to use it
Endpoint
— one specific "counter" in the app where a request goes in
Rate limit
— a "no more than X per person per Y minutes" rule
Commit
— one saved, labeled, undo-able change
Open redirect
— a link that looks safe but secretly sends you elsewhere
Fail-closed
— when unsure, default to locked, not open
The everyday version

Imagine the terminal never closes — foot and vehicle traffic never stops, 24/7. A routine dockside inspection turns up real problems while boats keep running: a free-sample cooler with no limit on how many bags one person can grab (someone could empty the whole thing before the next sailing); a "follow the detour" sign anyone could quietly re-point toward a scam parking lot; a suggestion-card slot that lets a stranger tape up a poisoned coupon; a single torn ticket that jams the whole printer; a dock clock stuck seven hours off so the sailing board shows the wrong time; and a "first person through the door gets to be dockmaster" rule with no ID check on a brand-new terminal. None of it requires shutting the terminal down to fix.

  1. Walk the dock and write down every real hole — not hypothetical ones, ones actually found by inspection. = the security audit that named all eight problems
  2. Patch each hole separately, one labeled logbook entry at a time, each with a witnessed test proving it's fixed — and never touch anything that already works. = one commit per fix, each with its own test; the audit-flagged fragile files stayed untouched
  3. Post sensible limits going forward — five free samples per person per ten minutes, not "unlimited" — so the same trick can't work twice. = rate limits + storage quotas + a fail-closed setup check
Pop quiz The free-sample cooler had no limit — what did a real visitor actually get to spam, unlimited, before this epic?
Anonymous uploads — anyone could submit scavenger-hunt photos (or survey answers, or analytics pings) with no cap at all. Enough scripted submissions could fill the server's entire 1 GB disk, which trips the health check and takes the whole site offline — not just the hunt page. The fix was a "5 uploads per 10 minutes per visitor" rate limit plus a hard storage quota, so one bad actor can dent the shelf but never empty the store.
Berth 2

The Boarding Checkpoint

E02 — Test floor + CI gates

Before this epic, any change to the app's code could sail straight onto the live site the moment someone pushed it — there was no inspection at all. This epic didn't fix a single bug. Instead, it built the checkpoint: automated checks that write down exactly how the app behaves today, a robot that tries every single staff-only door on the live app with no key to prove they're all still locked, an accessibility pass confirming the main pages still work for screen-reader visitors, and — the important part — a rule that no change can reach the live site until it clears every one of the above.

Terms, translated
Test
— a small automatic script that checks one thing works, and yells if it doesn't
CI
— a robot that re-runs every test automatically on every proposed change
Branch protection
— a rule that a change literally can't reach the live site until CI passes
Characterization test
— writing down what the code does today, not deciding what it should do
Accessibility smoke test
— an automatic check that screen-reader / keyboard-only visitors can still use it
The everyday version

Up to this point, terminal staff could walk any new procedure straight onto a departing ferry — no inspection, no manifest check, nothing. This epic installs the boarding checkpoint: a laminated card describing exactly how the current ferry runs today, board by board, so nobody "improves" something by accident; a guard who walks the entire terminal trying every door with no key, confirming all the staff-only and member-only doors are actually locked; a ramp check confirming a wheelchair could still get up the gangway on the six main public docks; and a chain across the gangway that simply will not lift until every one of those checks comes back clean. From here on, nothing boards without clearing the checkpoint first.

  1. Write down exactly how the ferry runs today — the doors, the schedule board, the busyness gauge — as a checklist that fails loudly the moment reality drifts from it. = characterization tests for the data layer, login/sessions, the ferry forecast model, and the site's editable-text registry
  2. Send a guard to rattle every staff-only and member-only door with no key and confirm each one holds; separately confirm the gangway is wheelchair-accessible on the public docks. = a generated test that hits every admin/portal route unauthenticated + an automated accessibility smoke test
  3. Bolt a chain across the gangway that only lifts once the guard's report and the schedule checklist both come back clean — no exceptions, not even for staff. = a required CI check + branch protection on the main line, so nothing reaches production unreviewed
Pop quiz The guard finds one admin-only door swinging open with no key. What does this epic's rulebook say the guard does next?
Stop and report it to the dockmaster — never fix it quietly, never wave it through. An actually-unlocked door would be a live security hole, and this epic's whole job is to detect and prove, never to silently patch or silently excuse a finding. As it happened, the guard walked every one of the 24-plus staff doors and 12-plus member doors on this app and found them all locked — so nothing needed handing off this time.
Berth 3

Alarms, a Vault, and a Considered Move

E03 — Ops floor + repo migration

This epic gave the terminal an actual operations desk. Two watchdogs now page a human the moment something breaks or goes dark. Every night, a copy of all the terminal's records gets scrambled with a lock that only one specific key (kept safely elsewhere) can open, and driven off-site — so a fire in the terminal itself can't destroy the only copy. A second, private practice dock went up — a working double of the real terminal — so risky changes get tried there first. And the team looked into moving the whole terminal to a different port, only to find the new port's slip already had someone else's boat tied up in it — so the move is on hold. Everything else shipped anyway, because none of it needed the move to happen first.

Terms, translated
Monitoring
— a service that watches your app and taps you on the shoulder when it breaks
Uptime check
— "is it on and responding right now," checked every few minutes
Encryption
— scrambling data with a secret key so only the matching key can read it
Staging
— a working practice copy of the real site, not shown to real visitors
Repo
— the master folder holding all the app's code plus its full history of changes
Token / secret
— a password-like string a robot uses instead of a human typing a real password
The everyday version

The terminal gets an operations desk. Two watchdogs go in: one barks the instant the ticket machine throws an error, one barks if the lights simply go dark. Every night, the day's paperwork gets copied, locked in a code only one specific key can open, and driven to a vault across town — so a fire at the terminal can't wipe out the only copy. A second, smaller practice dock goes up out back — same layout, same equipment, but nobody boards a real ferry there, so new procedures get rehearsed first. And the operator seriously considers relocating the whole terminal to a bigger port down the coast — only to discover someone had already tied a rowboat up in that exact slip and left it there. Rather than start that fight today, the move goes on the shelf. The watchdogs, the vault, and the practice dock get built anyway — none of them needed the new address.

  1. Install two watchdogs — one reports errors the instant they happen, one checks the terminal is even open, every five minutes. = server-side error monitoring (Sentry) + an uptime checker (UptimeRobot) on the health page and the live ferry feed
  2. Every night, copy the day's records, lock them with a key that never leaves a specific safe, and store the locked copy somewhere else entirely. = an automated nightly backup, encrypted before it ever leaves the server, stored in off-site cloud storage; the private key lives only in a password manager
  3. Build a practice dock for rehearsing risky changes, and separately weigh moving to a new port — proceed with the watchdogs and the vault regardless of what happens with the move. = a staging copy of the app for safe testing; a move to a different hosting account was evaluated, found blocked, and paused — everything else shipped anyway
Pop quiz Do the watchdogs, the vault, or the practice dock need the terminal to actually finish moving to the new port first?
No — none of them depend on the move at all. That's exactly why all three finished even after the move itself got paused. Monitoring, backups, and the practice dock attach to the terminal wherever it happens to be docked right now; only the "new address" paperwork — the actual account transfer — is the one thing sitting on the shelf, ready to pick back up if and when the slip-squatting rowboat gets sorted out.
How the three connect

Same terminal, three passes

E01 patched real holes in a terminal that could never close. E02 built the checkpoint so those holes — or new ones like them — can't slip back in by accident. E03 gave the whole operation a watch desk, a vault, and a rehearsal space, and quietly filed away a move that turned out not to be worth forcing today.

Fix what's broken → Guard the gate → Watch, back up, rehearse
